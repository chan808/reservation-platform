# REQUIREMENTS.md

작성일: 2026-03-11

## 프로젝트 목적

특정 시간에 오픈되는 한정 상품 판매, 티켓팅, KTX 예약과 유사한 상황을 가정한다.
핵심은 다음 두 가지다.

- 동시 접속 상황에서도 재고 정합성을 깨지지 않게 유지할 것
- 단계별 아키텍처 개선을 통해 처리량과 응답시간을 개선할 것

## 단계별 목표

| 단계 | 목표 | 상태 |
|------|------|------|
| 1차 | 단일 서버 + DB 락 기반으로 주문/결제/재고 정합성 완성, 병목 구간 측정 | 완료 |
| 2차 | Redis + Kafka 기반으로 재고 차감과 결제 이벤트를 분산 처리로 개선 | 다음 작업 |
| 3차 | 대기열, SSE, 다중 서버, 수평 확장 검증 | 이후 작업 |

## 1차 완료 범위

### 완료된 백엔드 기능

- 회원 템플릿 기반 인증 구조 유지
  - JWT
  - OAuth
  - 회원 탈퇴 soft delete
  - 탈퇴 시 이메일 / provider_id 익명화
- 상품 모듈 구현
  - 상품 등록
  - 상품 목록 / 상세 조회
  - 판매 상태 계산
- 주문 모듈 구현
  - 주문 생성
  - 주문 조회
  - 주문 취소
  - 주문 상태 이력 기록
  - 주문 만료 스케줄러
- 결제 모듈 구현
  - 결제 준비
  - 결제 승인
  - 결제 취소
  - 웹훅 저장 및 처리
  - 결제 전략 구조
- 재고 처리
  - DB 비관적 락 기반 차감
  - 결제 실패 / 취소 / 만료 시 재고 복구
- 스키마
  - Flyway 기반 상품 / 주문 / 결제 테이블 초안 완료
  - 회원 withdrawn_at soft delete 반영

### 1차 아키텍처 결정

- 주문 생성 시 상품은 한 트랜잭션 안에서 락 조회와 재고 차감을 같이 처리한다
- 결제 승인 후 주문 상태 반영은 동기 처리로 유지한다
- 회원 탈퇴는 withdrawn_at 기반 soft delete로 유지한다
- 이메일 재사용은 탈퇴 시 익명화로 허용한다

### 1차 부하 테스트 범위

- 기준 시나리오
  - constant
  - 동일 상품 대상 상품 조회 -> 주문 생성 -> 결제 승인 -> 주문 조회
- 보조 시나리오
  - burst
  - 0 -> 목표 VU 급증 -> 유지 -> 종료
- 정합성 검증
  - stock-check.js로 confirmed payments 수와 실제 재고 차감 수 일치 확인

- [x] 결제 승인 / 취소 / 웹훅 처리

## 1차 성능 결론

- constant 결과 기준, 500 ~ 1000 VU 구간부터 처리량 증가 폭이 급격히 줄어듦
- 1000 VU 이상에서 p95 / p99 및 flowP95 / flowP99 지연이 크게 증가함
- 재고 정합성은 유지됨
- 즉, 1차는 정합성은 확보했지만 중앙 DB 락 경합으로 포화가 명확히 드러난 상태다

## 2차 작업 범위

2차의 목적은 정합성을 유지하면서 DB 중심 병목을 줄이는 것이다.

### 2차 핵심 구현 항목

- Redis 재고 미러링
- Lua script 기반 원자적 재고 차감
- Redisson 또는 동등한 분산 제어로 중복 요청 억제
- Kafka 기반 주문 / 결제 이벤트 비동기 처리
- 결제 후속 처리의 이벤트 기반 전환
- 필요 시 outbox 패턴 도입

### 2차에서 반드시 할 일

- [ ] 상품 재고를 Redis에 반영하는 구조 설계
- [ ] DB -> Redis 재고 동기화 정책 결정
- [ ] Lua script로 재고 차감 / 복구 구현
- [ ] 중복 주문 / 멱등 키 처리 강화
- [ ] Kafka 토픽 / producer / consumer 설계
- [ ] 결제 성공 / 실패 후속 처리 비동기 전환
- [ ] 장애 시 재처리 / 보상 처리 기준 정리
- [ ] 기존 constant 시나리오로 1차와 동일 조건 재측정
- [ ] burst 시나리오로 오픈 순간 트래픽 재측정

### 2차 비교 기준

2차는 아래 조건을 1차와 동일하게 유지한다.

- 동일 상품
- 동일 주문 플로우
- 동일 paymentType
- 동일 orderQuantity
- 동일 테스트 시간
- 동일 VU 구간
- 동일 정합성 검증 방식
