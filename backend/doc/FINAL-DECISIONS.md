## 1. 1차 구현 완료 내용 요약

### 도메인 / 기능

- 회원 구조는 기존 템플릿을 유지하고, 주문 / 결제 시스템을 그 위에 연결했다
- `product`, `order`, `payment` 모듈을 분리했다
- 상품 조회, 주문 생성, 주문 조회, 주문 취소, 결제 준비, 결제 승인, 결제 취소, 웹훅 처리까지 1차 흐름을 완성했다
- 주문 타임아웃 시 자동 취소와 재고 복구를 구현했다
- 결제 실패 / 취소 / 만료 시 재고가 복구되도록 처리했다

### 데이터 / 정합성

- Flyway로 상품 / 주문 / 결제 관련 스키마를 추가했다
- 회원 탈퇴는 `members.withdrawn_at` 기반 soft delete로 확정했다
- 탈퇴 시 이메일과 `provider_id`는 익명화해서 재사용 가능하도록 했다
- 주문의 `member_id` FK는 유지했다
- 주문 생성 시 상품은 한 트랜잭션 안에서 락 조회와 재고 차감을 함께 처리하도록 정리했다
- 결제 승인 후 주문 상태 반영은 동기 처리로 유지했다

### 결제 구조

- 결제 수단은 전략 패턴으로 분리했다
- `TOSS`, `KAKAO`, `PAYPAL` 확장 구조를 만들었다
- 1차 부하 테스트는 자동화가 쉬운 `PAYPAL` 경로를 기준으로 사용했다

### 부하 테스트 / 운영성 도구

- `backend/perf` 안에 시드, 토큰 생성, 상태 초기화, k6 실행 래퍼, summary 확인 스크립트를 정리했다
- `constant` 시나리오와 `burst` 시나리오를 모두 지원한다
- `stock-check.js`로 성공 결제 수와 실제 재고 차감 수가 일치하는지 확인할 수 있게 했다
- 비교용 테스트에서는 재고를 크게, 품절 검증에서는 재고를 작게 줄 수 있도록 파라미터화했다
- VU 확장을 위해 perf 계정 수를 늘릴 수 있게 했고 현재 기본값은 `5000`이다

### 1차 성능 결론

- constant 기준으로 500 ~ 1000 VU 구간부터 처리량 증가가 둔화됐다
- 1000 VU 이상에서 전체 플로우 p95 / p99 지연이 크게 증가했다
- 성공률과 재고 정합성은 유지됐다
- 따라서 1차 구현은 정합성은 확보했지만 DB 락 경합으로 인해 확장 한계가 명확하다

## 2. 2차에서 할 작업 목록

### 목표

- DB 중심 병목을 줄이고, Redis + Kafka 기반 구조로 처리량과 지연을 개선한다
- 1차와 동일 시나리오로 재측정하여 개선 효과를 정량 비교한다

### 구현 우선순위

1. Redis 재고 구조 설계
2. Lua script 기반 원자적 재고 차감 / 복구 구현
3. 중복 요청 / 멱등 키 / 재시도 정책 강화
4. Kafka 기반 주문 / 결제 이벤트 비동기 처리
5. 필요 시 outbox 패턴 도입
6. 장애 시 보상 처리 및 재처리 기준 정리

### 세부 작업

- [ ] 상품 재고를 Redis에 적재하는 방식 결정
- [ ] Redis와 DB 사이 동기화 정책 결정
- [ ] Lua script로 재고 차감 / 복구 구현
- [ ] 주문 생성의 중복 요청 억제 방식 정리
- [ ] Redisson 또는 동등한 분산 제어 도입 여부 확정
- [ ] Kafka 토픽 구조 설계
- [ ] producer / consumer 구현
- [ ] 결제 후속 처리의 이벤트 기반 전환
- [ ] 실패 이벤트와 재고 복구의 비동기 경로 정리
- [ ] constant 시나리오 재측정
- [ ] burst 시나리오 재측정
- [ ] 1차 대비 TPS / p95 / p99 / flowP99 / 정합성 비교표 작성
