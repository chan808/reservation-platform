## VU 1명이 반복하는 작업
1. 자기 계정 1개를 사용(tokens.csv에서 자기 토큰 1개 받음, 로그인 api는 rate limit 문제로 패스)
2. 상품 조회
    - GET /api/products/{productId}
    - 상품이 ON_SALE인지 확인
    - 재고가 음수인지 체크
3. 주문 생성 요청을 보냄
    - POST /api/orders
    - payload:
        - productId
        - quantity 보통 1
        - orderRequestId 매 반복마다 새 값
        - paymentType 보통 PAYPAL
    - 여기서 재고 차감 경쟁이 실제로 발생
4. 주문 생성이 성공하면 결제 승인을 보냄
    - POST /api/orders/{orderId}/confirm-payment
    - payload:
        - paymentKey
        - amount
5. 결제 승인 성공 후 주문 최종 상태를 조회
    - GET /api/orders/{orderId}
    - 주문 상태가 PAID
    - 결제 상태가 SUCCEEDED 인지 확인
6. 위 4단계가 전부 성공하면 completed flow 1건으로 집계
7. 그 다음 바로 다시 2번부터 반복
    - 기본 설정은 think time 0초라서 계속 반복.
한 VU가 사람처럼 “상품 보고 -> 주문 -> 결제 -> 결과 확인”을 계속 반복

## 지속 부하, 집중 부하 테스트 차이
- 지속 부하 테스트는 정해진 수의 사용자가 테스트 시간 내내 계속 붙어 있는 시나리오. 구조의 평균 처리량, 안정성, 포화 구간을 비교.
- 집중 부하 테스트는 짧은 시간에 사용자가 한꺼번에 몰렸다가 유지되고 빠지는 시나리오. 오픈 순간 예약 폭주나 한정 상품 판매처럼 순간 유입 대응력 확인
